# NorthStark Analytics - Sales & Customer Performance Reporting

This project is my response to the DSC NCAP application scenario. Rather than answering the question in writing alone, I simulated the scenario as a real working project to show how I would actually approach it.

The scenario describes joining an active consulting project midway, finding messy data across multiple source systems, unclear metric definitions, and incomplete documentation, with a client update due by end of day. This repository walks through exactly how I handled that, from raw data to cleaned metrics to a working dashboard that shows accurate data.


## About the Data

The data was generated by Claude (Anthropic) to replicate the scenario as realistically as possible. It simulates two separate source systems that a real SaaS company like Northstark Analytics would use, along with a third file tracking customer cancellations.

**crm_customers.csv** — The CRM system

This file represents the customer records managed by the sales team. It contains customer IDs, company names, industries, cities, subscription plans, pipeline stages, signup dates, and assigned account managers. This is the system of record for who the customers are and where they are in the sales process.

**billing_transactions.csv** — The Billing system

This file represents every financial transaction processed by the billing platform. Each row is a monthly subscription charge, with the transaction ID, customer ID, amount, date, status, and plan. This is the system of record for revenue.

**churn_data.csv**

This file contains a list of customers who cancelled their subscriptions, along with the cancellation date and reason. It is used to determine which customers are still active and to calculate churn rate.
 
## Medallion Architecture
 
This project follows the Bronze and Silver layers of the medallion architecture.
 
**Bronze** is the raw data layer. The three source CSVs are loaded into Spark as-is, with only the date format parsed so Spark can work with them. No business logic is applied. Bronze is the audit trail and the source of truth for what came in from each system.
 
**Silver** is the cleaned and transformed layer. This is where all business logic lives. Customer IDs are reconciled across systems, duplicates are removed, refunds are excluded, pipeline stages are standardized, and the final metrics are calculated. Silver data is what powers the dashboard.
 
## SQL Transformations
 
All transformations from Bronze to Silver are written in PySpark SQL and live in `src/silver.py`. The following techniques are used:
 
- **CTEs** to build each transformation step by step before joining everything together
- **ROW_NUMBER() window function** to identify and remove duplicate billing rows
- **LEFT JOIN** to reconcile customer IDs across the CRM and billing tables
- **CASE WHEN** to standardize inconsistent pipeline stage values
- **Primary and foreign key relationships** documented in the schema to make table relationships explicit

## Data Quality Issues Found
 
Exploratory data analysis was performed across all three source files before any metrics were calculated. The EDA notebook can be found in the `notebook/` folder. The following issues were identified and resolved:
 
| Issue | Source File | What Was Done |
|---|---|---|
| Customer IDs in two different formats (CUST-001 in CRM, C001 in billing) | CRM + Billing | Standardized all IDs to CUST-XXX format before joining |
| Inconsistent date formats (YYYY-MM-DD vs DD/MM/YYYY ) | CRM | Parsed using PySpark coalesce and to_date with multiple format patterns |
| Pipeline stage values inconsistent (e.g. closed_won, Won, QUALIFIED, null) | CRM | Mapped all variants to a standard set of stage names using CASE WHEN in SQL. Nulls assigned to Lead |
| Duplicate transaction rows | Billing | Removed using ROW_NUMBER() window function, partitioned by transaction_id |
| Refunds included as negative amount rows mixed into revenue | Billing | Filtered out using WHERE status = 'paid' before aggregating revenue |
| 10 customers present in billing but missing from CRM | Billing | Flagged as an open item. Included in customer count but noted for client review |
 
Cleaned versions of all data are written to the `data/silver/` folder.
 
## Key Metrics
 
These figures are calculated from Silver data only and represent what would go into the client update.
 
| Metric | Definition |
|---|---|
| Total Revenue | Sum of paid transactions only. Refunds and duplicates excluded. |
| Active Customers | Unique customers with at least one paid transaction who have not churned |
| Churn Rate | Churned customers divided by total unique customers in billing |
 
## Assumptions
 
Every assumption made during cleaning and metric calculation is documented here. In a real engagement these would be shared with the client for confirmation before the final report.
 
1. Revenue is defined as paid transactions only. Refunds are excluded.
2. A customer is considered active if they have at least one paid transaction and do not appear in the churn list.
3. Churn rate is calculated as churned customers divided by total unique customers found in the billing system.
4. Null pipeline stages in the CRM are treated as `Lead`, the earliest stage in the funnel.
5. `CUST-001` in the CRM and `C001` in billing refer to the same customer. The numeric portion of the ID is the shared key.
6. The 10 customers found in billing but not in the CRM are included in revenue and customer counts but flagged as an open item for the client to investigate.
